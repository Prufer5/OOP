\section{Introduzione}

Quando si programma nel piccolo, è il singolo utente a usufruire di tale codice (o comunque poche persone) che quindi hanno sotto controllo tutto il programma ed è più difficile creare fraintendimenti.
Quando si programma nel grande, bisogna avere certe accortezze in modo che tutti possano usufruire del codice.
È molto facile avere problemi di fraintendimento, overhead e incertezza relativa alle parti di terzi.
L'overhead si riferisce alle risorse (tempo di esecuzione, memoria) aggiuntive necessarie per l'esecuzione del codice a causa dei meccanismi specifici dell'OOP, come l'ereditarietà, il polimorfismo e la gestione dinamica della memoria.
Un modo per aiutare  è la documentazione.
L'UML va molto bene per capire, progettare e comunicare e per singole parti del design, ma è difficile usarlo come unica rappresentazione di tutta la documentazione.
Inoltre è è inevitabile che occorra modificare il software nel caso di cambi di requisiti.
E quindi, per ogni modifica:
\begin{itemize}
    \item l'overhead di comunicazione cresce
    \item la probabilità di fraintendimenti anche
\end{itemize}



\section{Astrazioni}

L'obiettivo è di creare moduli che interagiscono tra di loro in modi semplici e ben definiti.
La \textbf{decomposizione} la si effettua per suddividere un problema in sottoproblemi (figura \ref{fig:Decomposizione}) così che:
\begin{itemize}
    \item si abbiano dettagli allo stesso livello,
    \item possano essere risolti indipendentemente 
    \item soluzioni a sottoproblemi possono essere combinati per ottenere la soluzione del problema originario
\end{itemize}
\begin{figure}
    \centering
    \includegraphics[width=0.5\linewidth]{Decomposizione.png}
    \caption{Esempio di decomposizione}
    \label{fig:Decomposizione}
\end{figure}
Esempio: 
\begin{lstlisting}
    public int Factorial(int n)
    {
        if(n == 0)
        {
            return 1;   // sottoproblema 1
        }
        else
        {
            return n * Factorial(n - 1);    // sottoproblema 2
        }
    }
\end{lstlisting}
Abbiamo creato l'astrazione \verb|Factorial|, specificata dal suo nome, i parametri di input, il tipo di ritorno ecc
Il suo contratto è "if n $>=$ 0 then the output y = n!”.
Attraverso questo abbiamo decomposto il problema in sottoproblemi.
Con l'astrazione ignoriamo certi dettagli con lo scopo di semplificare il problema originario.
L'astrazione è un modo per decomporre un problema.
Due meccanismi fondamentali:
\begin{itemize}
    \item \textbf{Abstraction by parametrization}: con questa vogliamo ottenere generalizzazione e riutilizzo di codice con dati diversi.
    \item \textbf{Abstraction by specification}: rimuoviamo i dettagli di implementazione (come) e definiamo un contratto: ti prometto qualcosa se tu mi dai qualcosa in cambio.
\end{itemize}


\subsection{Parametrization}


Esempio: 
Al posto di scrivere il sort così:
\begin{lstlisting}
    public static void main(String a[])
    {
        int[] arr1 = {10, 34, 2, 56, 7, 67, 88, 42};
        for(int i = 0; i < arr.length; i++)
        {
            key = arr[0];
            j = i - 1;

            while(j >= 0 && arr[j] > key)
            {

\end{lstlisting}
Usiamo la parametrization e scriviamo così:

    }

    {

        {

            {
                arr[j + 1] = arr[j];
                j--;
            }

            arr[j + 1] = key;
        }
    }
\end{lstlisting}


\subsection{Specification}


Un contratto è una specifica formale (o informale) che definisce cosa una classe, metodo o interfaccia deve garantire in termini di:
\begin{itemize}
    \item \textbf{Precondizioni}: cosa deve essere vero prima dell'esecuzione (required)
    \item \textbf{Postcondizioni}: cosa deve essere vero dopo l'esecuzione (return e modify)
    \item \textbf{Invarianti}: cosa deve rimanere sempre vero per l'oggetto.
\end{itemize}
Se la precondizione è vera, allora la postcondizione è garantita, altrimenti tutto può succedere.
\begin{lstlisting}
    public static void main(String a[])
    {
        int[] arr1 = {10, 34, 2, 56, 7, 67, 88, 42};
        arr = insertionSort(arr);
    }

    /**
    * MODIFY the array a so that values are ordered, increasing
    * @param a items to be sorted, increasingly
    * @return the modified array
    */

    private static void insertionSort(int[] arr)
    {
        for(int i = 0; i < arr.length; i++)
        {
            key = arr[0];
            j = i - 1;

            while(j >= 0 && arr[j] > key)
            {
                swap(arr, j);
                j--;
            }

            arr[j + 1] = key;
        }

        return arr;
    }

    /**
    * swap a[j] with a[j - 1]; MODIFY the array a
    * @param a, REQUIRED to have 2 or more elements
    * @param j an index of the array, REQUIRED to be a valid index and > 0.
    */

    private static void swap(int[] arr, int j) 
    {
        int temp;
        temp = arr[j];
        arr[j] = arr[j - 1];
        arr[j-1] = temp;
    }
\end{lstlisting}


\subsection{Decomposition vs specification}


La decomposizione consiste nel suddividere un problema complesso in sottoproblemi più semplici, più gestibili e più comprensibili. 
È un processo di tipo top-down, utile per progettare sistemi modulari.
La specificazione è il processo di descrivere cosa deve fare un componente o sistema, senza dire come lo farà. 
Specifica i requisiti funzionali e le interfacce.


\subsection{Procedural abstraction}


Si definiscono delle procedure/funzioni per estendere il linguaggio con nuove operazioni.
Non nascondono i dettagli dell'implementazione e potrebbero essere indipendenti.
